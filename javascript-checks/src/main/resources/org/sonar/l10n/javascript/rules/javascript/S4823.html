使用命令行参数是安全敏感的。它在过去导致了以下漏洞:

  • CVE-2018-7281
  • CVE-2018-12326
  • CVE-2011-3198

与任何其他用户输入一样，命令行参数也可能是危险的。在未经验证和过滤的情况下，不应该使用它们。

还请记住，任何用户都可以检索在系统上运行的进程列表，这将使提供给它们的参数可见。 因此，通过命令行参数传递敏感信息应该被认为是不安全的。

当在每个程序入口点(main methods)上使用命令行参数时，该规则都会引发一个问题。目标是指导安全代码审查。

问问你自己是否

  • 使用任何命令行参数都不需要先进行过滤。
  • 应用程序通过命令行参数接受敏感信息。

如果你对这些问题中的任何一个回答是肯定的，那么你就处于危险之中。

推荐的安全编码实践

过滤 所有命令行参数在使用之前。

任何用户或应用程序都可以列出正在运行的进程，并查看它们开始时的命令行参数。与在命令行中公开敏感信息相比，向应用程序提供敏感信息有更安全的方法。 通常将它们写入进程的标准输入，或者给出包含这些信息的文件的路径。

有问题的代码示例

// process对象是一个全局对象，它提供关于当前Node.js进程的信息，并对其进行控制
var param = process.argv[2]; // 有问题的: 检查这个参数是如何使用的
console.log('Param: ' + param);
